# Производственный план Snow Sling

Версия: 1.0  
Дата: 18 июля 2026 года  
Связанное ТЗ: `TZ_Snow_Sling_Web_Game.md`

## 1. Моё обязательство как разработчика

Моя цель — не выдать как можно больше файлов за один ответ, а довести игру до состояния, в котором:

- сам запуск и езда приятны без наград и красивого меню;
- управление соответствует жесту игрока;
- трасса предлагает решения, а не только декорации;
- графика выглядит как единый продукт, а не коллекция случайных ассетов;
- прогрессия ощутима и не маскирует слабый геймплей;
- игра стабильно работает на реальном мобильном браузере;
- качество подтверждено тестами, скриншотами, браузерными заездами и измерениями производительности.

Первая реализация любой сложной механики считается гипотезой. Она становится готовой только после проверки и, при необходимости, нескольких итераций настройки.

## 2. Решение по инструментам

### 2.1. Codex — основной разработчик

Codex будет использоваться для полного цикла разработки:

- чтение и поддержание архитектуры проекта;
- написание TypeScript, CSS, шейдеров и тестов;
- создание конфигурации сборки;
- запуск приложения и автоматических проверок;
- управление Playwright-сценариями;
- анализ скриншотов, console errors и профилей;
- исправление дефектов;
- контроль diff и регрессий;
- ведение документации и чек-листов.

Codex не заменяет игровой движок. Он управляет разработкой и инструментами. Добавлять OpenAI API непосредственно в runtime игры для «самогенерации игры» не нужно: это увеличит стоимость, задержку и количество точек отказа, но не улучшит физику или графику готовой сборки.

Чтобы работа не деградировала между длинными этапами:

- требования хранятся в репозитории;
- постоянные правила находятся в `AGENTS.md`;
- каждый этап имеет вход, выход и quality gate;
- решения фиксируются в коротком журнале;
- проверки выполняются командами, а не по памяти.

### 2.2. Основной runtime: Three.js + TypeScript + Vite

Выбор обусловлен задачей:

- процедурная трасса и собственная геометрия;
- фиксированная симуляция вне renderer;
- полный контроль update/render loop;
- лёгкая автоматизация через обычные файлы и Git;
- удобные unit/E2E-тесты;
- GLB/glTF, Meshopt/Draco, KTX2;
- прямой контроль object pooling, instancing и floating origin;
- отсутствие зависимости от облачного редактора.

Используются:

- Three.js для сцены и WebGL;
- DOM/CSS для HUD и меню;
- собственная surface-following симуляция для санок;
- Rapier только при доказанной необходимости для второстепенных физических объектов;
- Vitest для правил и генератора;
- Playwright для управления и визуального QA;
- Blender/glTF Transform для production-ассетов.

### 2.3. PlayCanvas — сильный, но не основной выбор

PlayCanvas подходит для профессиональной браузерной 3D-разработки. Актуальная платформа предлагает:

- open-source PlayCanvas Engine;
- WebGL2 и WebGPU;
- npm и tree-shakable ES modules;
- визуальный Editor;
- импорт glTF/GLB;
- Draco/Meshopt и сжатие текстур;
- встроенный profiler;
- live update на мобильном;
- version control внутри Editor;
- VS Code extension;
- MCP-интеграцию редактора.

Однако для Snow Sling есть риски:

- значительная часть мира создаётся процедурно в runtime, поэтому визуальный scene editor даёт меньше преимущества;
- собственная фиксированная симуляция и chunk streaming всё равно потребуют большого объёма кода;
- облачный Editor создаёт второй контур состояния рядом с Git-репозиторием;
- автоматическое тестирование и воспроизводимые headless-проверки проще держать в обычном file-first проекте;
- использование Editor API как основы автоматизации рискованно, потому что официальная документация помечает его beta;
- смена движка после начала production-контента будет дорогой.

**Решение:** основной проект создаётся на Three.js. PlayCanvas Engine и Three.js не смешиваются. PlayCanvas рассматривается только в двух случаях:

1. пользователь прямо выбирает PlayCanvas до начала реализации;
2. ограниченный spike на отдельной ветке доказывает измеримое превосходство: быстрее импорт/настройка ассетов, не хуже автоматизация, сборка и FPS, без привязки критической логики к Editor.

Если spike проводится, он использует одну сцену, одного персонажа, один чанк и одно мобильное измерение. После выбора альтернативная реализация удаляется из производственного пути. Нельзя поддерживать две версии игры.

Официальные источники PlayCanvas для повторной проверки перед решением:

- https://playcanvas.com/products/engine
- https://playcanvas.com/products/editor
- https://developer.playcanvas.com/user-manual/editor/version-control/
- https://developer.playcanvas.com/user-manual/editor/scripting/vscode-extension/
- https://developer.playcanvas.com/user-manual/editor/editor-api/

### 2.4. Figma

Figma-плагин установлен и применяется для:

- визуального направления UI;
- макета базы;
- HUD в состояниях low/high speed;
- результата и прокачки;
- адаптации 360×640, 390×844, tablet и desktop;
- проверки safe area;
- передачи размеров, токенов и состояний DOM-реализации.

Figma не является источником игровой логики. Макет должен проверяться поверх реального скриншота трассы, иначе можно создать красивый, но мешающий HUD.

### 2.5. Blender и asset pipeline

Blender используется для:

- очистки и настройки моделей;
- проверки масштаба, pivot, facing и ground contact;
- рига и анимаций;
- collision proxies;
- LOD;
- экспорта GLB;
- при необходимости запекания простого освещения.

После Blender:

1. GLB валидируется;
2. glTF Transform удаляет лишнее и объединяет данные;
3. сравниваются Meshopt/Draco;
4. текстуры переводятся в KTX2 при реальной выгоде;
5. ассет проверяется в браузере на целевом устройстве;
6. фиксируются triangle/material/texture/size budgets.

### 2.6. GitHub

Установленный GitHub-плагин нужен не для написания механики, а для дисциплины поставки:

- создание понятных checkpoint-коммитов;
- review diff;
- CI;
- анализ упавших GitHub Actions;
- draft PR перед крупным объединением;
- сохранение истории решений.

Публикация, push и PR выполняются только когда репозиторий подключён и пользователь просит это сделать. До этого локальные файлы остаются источником истины.

## 5. Постоянный цикл работы на каждом этапе

Каждая задача проходит один и тот же цикл:

1. **Inspect.** Прочитать текущее состояние, ближайший код, тесты, TODO и предыдущий отчёт.
2. **Define.** Сформулировать одно проверяемое изменение и критерий готовности.
3. **Implement.** Сделать минимально достаточное изменение без параллельной переделки несвязанных систем.
4. **Static verify.** Format, lint, typecheck, unit tests, build.
5. **Play.** Запустить игру, выполнить реальные игровые действия короткими сериями с паузами.
6. **Observe.** Проверить скриншоты, визуальное состояние, консоль, FPS и input response.
7. **Stress.** Если затронута горячая система, выполнить длительную/массовую проверку.
8. **Review.** Просмотреть diff как критичный reviewer: баги, регрессии, временные хаки, утечки.
9. **Decide gate.** PASS или FAIL с фактами.
10. **Checkpoint.** Обновить краткий журнал, тесты и документацию; только затем следующий этап.

Если quality gate = FAIL, новый контент не добавляется. Исправляется причина провала.

## 6. Этапы производства

### Этап 0. Создание производственного контура

**Цель:** сделать так, чтобы проект технически не мог тихо деградировать.

Работы:

- создать Git-структуру и понятные scripts;
- добавить `AGENTS.md`, план и ADR для ключевых решений;
- настроить TypeScript strict, lint, format, Vitest и Playwright;
- создать debug/performance overlay;
- настроить сохранение скриншотов и отчётов playtest;
- определить device/viewports matrix;
- создать asset manifest и бюджеты;
- настроить LocalPlatformBridge;
- зафиксировать seed и input recording для воспроизводимых заездов.

Артефакты:

- запускаемый пустой shell;
- CI-ready команды;
- тест первого экрана;
- папка `artifacts/playtests/<date-stage>`;
- baseline загрузки и FPS пустой сцены.

Quality gate G0:

- clean install/build выполняется;
- первая сцена открывается без console errors;
- portrait resize и pause работают;
- Playwright сохраняет скриншот;
- debug overlay показывает frame time.

### Этап 1. Серый прототип запуска и езды

**Цель:** доказать удовольствие от главного действия без графических украшений.

Работы:

- фиксированная симуляция 60 Гц;
- поверхность с несколькими уклонами;
- forward/lateral speed;
- сопротивление, гравитация, поворот;
- жест оттягивания и отпускания рогатки;
- мобильное relative-drag управление;
- естественная остановка;
- быстрый restart;
- input tape и deterministic replay.

Итерации настройки:

1. сила и читаемость запуска;
2. latency и dead zone;
3. скорость перекладки руля;
4. потеря скорости на повороте;
5. продолжительность раннего заезда;
6. поведение на 30/60/120 render FPS.

Quality gate G1 — самый важный:

- ездить интересно минимум 10 попыток даже на примитивах;
- нейтральный жест не ведёт к объекту;
- один input tape даёт одинаковый результат при разных render FPS;
- игрок понимает запуск без текстовой инструкции;
- остановка плавная и логичная;
- нет рывков камеры/симуляции.

Пока G1 не пройден, запрещены магазин, реклама, биомы и production-персонаж.

### Этап 2. Камера и game feel

**Цель:** превратить корректное движение в эмоционально приятное.

Работы:

- chase camera с look-ahead;
- speed-dependent FOV;
- visual lean отдельно от физики;
- следы и снег из пула;
- ограниченные camera impulses;
- базовые звуки launch/slide/wind;
- reduced motion;
- debug camera/speed presets.

Quality gate G2:

- низкая, средняя и высокая скорость визуально различимы;
- путь читается минимум на 1.5–2 секунды вперёд;
- камера не укачивает и не пересекает поверхность;
- частицы не закрывают препятствия;
- настройки reduced motion реально уменьшают движение.

### Этап 3. Бесконечный мир

**Цель:** обеспечить технический фундамент дальних заездов.

Работы:

- ChunkManager и deterministic seed;
- шесть серых шаблонов;
- safe corridor validator;
- 2 чанка позади, 7 впереди, подготовка следующих;
- object pools и instancing;
- floating origin;
- fallback chunk;
- тест швов, bounds и массовой генерации;
- при необходимости worker для данных, но только после профиля.

Quality gate G3:

- 10 000 логических чанков проходят валидатор;
- 15-минутный автоматизированный заезд не показывает швов и дрожи;
- cumulative distance не скачет при origin shift;
- память выходит на плато;
- генерация не создаёт серии кадров >50 мс;
- один шаблон не повторяется подряд.

### Этап 4. Игровые ситуации

**Цель:** сделать трассу игрой, а не красивым коридором.

Работы:

- снег, лёд, рыхлый снег;
- монеты и цепочки;
- 3 типа трамплинов;
- полёт, посадка и perfect landing;
- мягкие, средние и жёсткие столкновения;
- 8–12 шаблонов ситуаций;
- развилки риск/награда;
- телеграфы опасности;
- честные collision proxies.

Quality gate G4:

- каждые 2–4 чанка есть осмысленный выбор;
- на каждом участке один честный маршрут;
- рискованный путь даёт измеримую выгоду;
- монетные линии не ведут в неизбежный удар;
- зона посадки корректна на диапазоне скоростей;
- столкновение выглядит соответствующим визуальному контакту.

### Этап 5. Vertical slice графики

**Цель:** один короткий участок должен выглядеть как почти готовая игра.

Сначала утверждается art bible:

- оригинальный герой;
- формы и пропорции;
- палитра;
- материалы;
- уровень детализации;
- UI language;
- освещение и VFX;
- asset budgets.

Пайплайн:

1. с помощью imagegen создаются 2–3 направления, а не десятки случайных картинок;
2. пользователь выбирает направление;
3. персонаж и ключевые санки моделируются/подбираются;
4. asset skill проверяет GLB;
5. animation skill проверяет клипы и blends;
6. Blender inspection проверяет scale/facing/feet/ground;
7. ассет интегрируется в реальную сцену;
8. performance budget измеряется.

Quality gate G5:

- персонаж не является набором примитивов;
- силуэт читается на 360 px;
- анимации не snap и совпадают с состояниями;
- один биом визуально целостен;
- нет случайной смеси стилей;
- production-кадр достигает целевого FPS;
- размер обязательной загрузки остаётся в бюджете.

### Этап 6. Цикл результата, экономика и сохранение

**Цель:** создать причину для повторного заезда без эксплуатации игрока.

Работы:

- STOPPING/RESULTS/BASE;
- награда по источникам;
- три базовых улучшения;
- визуальная и физическая разница уровней;
- atomic purchase;
- SaveV1 и миграции;
- локальное/облачное слияние через интерфейс;
- защита от двойного начисления;
- ранняя кривая первых 15–25 заездов.

Quality gate G6:

- каждое из трёх улучшений ощущается в следующей попытке;
- игрок покупает первое улучшение после 1–2 заездов;
- нет отрицательной валюты/двойного списания;
- повреждённый save восстанавливается безопасно;
- мета не превращает плохой заезд в долгий grind;
- хороший навык даёт заметное преимущество при тех же уровнях.

### Этап 7. UI/UX в Figma и DOM

**Цель:** профессиональный интерфейс, который помогает, а не закрывает игру.

Работы:

- Figma-макеты поверх реальных игровых скриншотов;
- база, HUD, результаты, магазин, пауза, настройки;
- состояния affordable/locked/pressed/new;
- design tokens и CSS variables;
- RU/EN;
- safe area и landscape;
- reduced motion и размер UI;
- onboarding первых трёх заездов.

Проверка:

- 360×640;
- 390×844;
- 412×915;
- tablet portrait;
- desktop portrait-like window;
- landscape mobile.

Quality gate G7:

- центр и нижняя середина playfield свободны;
- постоянный HUD занимает минимальную площадь;
- все touch targets ≥44 CSS px;
- текст читаем на движущемся фоне;
- меню блокирует игровой input;
- первый запуск понятен без длинных инструкций.

### Этап 8. Звук и полировка feedback

**Цель:** каждое важное действие должно звучать и ощущаться.

Работы:

- слои снег/лёд/рыхлый снег;
- скорость-зависимый ветер;
- натяжение и отпускание;
- приземления и столкновения;
- pickup/combo/record/purchase;
- музыка базы и заезда;
- Web Audio unlock;
- pause/background/reklama lifecycle;
- vibration с настройкой.

Quality gate G8:

- поверхность различима с закрытыми глазами;
- звуки не наслаиваются бесконечно;
- после background/ad нет зависшего аудио;
- feedback сильный, но не утомляет после 10 минут;
- выключатели музыки/эффектов сохраняются.

### Этап 9. Контент

**Цель:** расширять подтверждённый фундамент, а не скрывать его недостатки.

Работы:

- 3 биома релиза;
- 12+ шаблонов;
- задания;
- типы санок;
- два слота модулей;
- косметика;
- постепенные открытия;
- баланс разнообразия.

Quality gate G9:

- биом меняет хотя бы одну игровую переменную, а не только цвет;
- новые санки создают стиль игры, а не прямую покупку победы;
- шаблоны не повторяются заметными сериями;
- первый час имеет открытия без перегрузки системами;
- косметика не влияет на физику.

### Этап 10. Яндекс Игры и монетизация

**Цель:** интеграция платформы не должна ломать одиночную игру.

Работы:

- актуальный Yandex SDK;
- readiness/gameplay markup;
- язык;
- Player data;
- leaderboard;
- rewarded/fullscreen ads;
- platform pause/audio lifecycle;
- offline/local fallback;
- лимиты и debounce облачных вызовов.

Quality gate G10:

- игра работает без SDK и без сети;
- реклама не движет симуляцию в фоне;
- награда выдаётся один раз;
- ошибка рекламы не отнимает базовую награду;
- cloud conflict не дублирует валюту;
- leaderboard failure не блокирует прогресс;
- актуальные требования модерации проверены перед релизом.

### Этап 11. Оптимизация

**Цель:** убрать реальные узкие места после профилирования.

Работы:

- CPU/GPU profile;
- draw calls/triangles/materials;
- texture memory;
- JS heap и GC;
- adaptive DPR/quality;
- LOD/instancing/batching;
- shader variants;
- начальная загрузка;
- context loss/recovery.

Правило: не проводить абстрактную оптимизацию до измерения. Каждый change должен иметь «до/после».

Quality gate G11:

- средний целевой Android: цель 60, минимум 45 FPS;
- 1% low не ниже 30 FPS;
- длительный заезд без устойчивого роста heap;
- обязательная загрузка укладывается в бюджет ТЗ;
- context loss восстанавливается или показывает безопасный fallback;
- auto quality не пульсирует.

### Этап 12. Release candidate

**Цель:** доказать готовность, не добавляя новые системы.

Работы:

- полный regression pass;
- первые 30 минут нового игрока;
- несколько clean profiles;
- browser/device matrix;
- отсутствие сети;
- background/rotate/resize;
- ad errors;
- save migration;
- тысячи seed;
- финальный review diff;
- карточка и материалы Яндекс Игр.

Quality gate G12:

- все обязательные тесты проходят;
- отсутствуют P0/P1 дефекты;
- P2 перечислены и не ломают основной цикл;
- production build без критических warnings;
- есть доказательства плейтеста и performance;
- Definition of Done из полного ТЗ выполнен.

## 7. Участие пользователя

Чтобы не заставлять пользователя утверждать каждую техническую мелочь, обязательны только три творческие контрольные точки:

1. **После G1:** нравится ли ощущение запуска, поворота и замедления.
2. **Перед G5:** выбор одного из 2–3 направлений героя и art style.
3. **После G7/G6:** оценка интерфейса и темпа прокачки.

Во всех остальных случаях разработчик самостоятельно принимает обратимые технические решения в рамках ТЗ и сообщает доказательства.

Если пользователь говорит «плохо управляется», это P0 для проекта независимо от количества готового контента.

## 8. Классификация дефектов

- **P0:** игра не запускается, прогресс теряется, управление действует против ввода, обязательный путь непроходим, реклама ломает награду.
- **P1:** частые просадки, нечестные столкновения, плохая камера, заметные швы, заезд скучен или апгрейд не ощущается.
- **P2:** локальная визуальная ошибка, редкий clipping, неидеальная анимация UI.
- **P3:** косметическое улучшение без влияния на понятность.

P0/P1 блокируют следующий quality gate. P2 допускается только с записью в backlog. P3 не должен вытеснять работу над ощущениями.

## 9. Что запрещено считать прогрессом

- увеличение числа файлов;
- добавление десяти уровней на одной и той же пустой трассе;
- примитивный персонаж, названный «готовой 3D-моделью»;
- меню, закрывающее слабую механику;
- десятки валют и кнопок;
- красивые статичные скриншоты без нормального движения;
- build success без запуска в браузере;
- 60 FPS на мощном desktop без мобильной проверки;
- случайная генерация без валидатора;
- «исправлено» без воспроизведения старого бага;
- реклама и мета до подтверждения core loop.

## 10. Формат отчёта после каждого этапа

```text
Этап: Gx — название
Статус: PASS / FAIL

Реализовано:
- ...

Проверено:
- команда/сценарий → результат
- viewport/device → результат
- screenshot/video/profile → путь к артефакту

Метрики:
- FPS/frame time
- draw calls/triangles/heap, если применимо

Оставшиеся проблемы:
- P0/P1/P2/P3 — описание и воспроизведение

Решение:
- почему этап можно/нельзя закрыть
- следующий ограниченный шаг
```

## 11. Финальное правило

Если приходится выбирать между «добавить новую функцию» и «довести уже существующее управление, камеру, трассу или производительность», выбирается доведение основы. Хорошая небольшая игра лучше большой плохой игры.
